# 版本迭代规则

## 目标

在修改 `stockTrade` 系统时，主动维护系统版本号，避免主版本源遗漏更新，或当前仍在使用的版本入口彼此不一致。

## 执行模式

本规则采用轻量模式。

轻量模式要求：

- 必须维护一个主版本源
- 默认只强制更新主版本源
- 仅在本次改动会影响对外展示、接口描述或运行时识别时，再同步其他有效入口
- 不默认回改历史设计文档、旧计划文档、归档说明

## 主版本源

系统正式版本号以以下文件为准：

- `stockTrade/backend/app/__init__.py`

其中的 `__version__` 是唯一主版本源。

## 版本号格式

系统版本号统一使用语义化版本格式：

- `MAJOR.MINOR.PATCH`
- 示例：`1.3.0`、`1.3.1`

不要使用日期版、四段版，或其他自定义主版本格式作为正式版本号。

## AI 必做检查

当你修改了以下内容时，必须检查是否需要升级版本号：

- 新增用户可感知功能
- 修复影响使用结果的缺陷
- 修改对外接口行为或返回口径
- 调整部署方式、运行方式、关键配置口径

以下改动默认不强制升级版本号：

- 纯注释修改
- 纯格式调整
- 无行为变化的重构
- 仅内部草稿文档更新

## 升级规则

- `patch`：Bug 修复、兼容性修正、用户可感知的小范围修正
- `minor`：新增功能、页面、接口、配置项
- `major`：不兼容变更、重大架构调整、需要迁移或适配的变化

如果用户没有明确指定新版本号，你应根据改动性质自行判断升级级别，并在最终说明中简要说明原因。

## Git Commit Message 规则

如果本次变更触发版本升级，且用户明确要求创建 release commit，则提交信息固定为：

- `chore(release): bump version to X.Y.Z`

其中 `X.Y.Z` 必须与主版本源一致。

如果本次只是普通功能开发、修复或文档修改，不使用 release commit message，继续沿用 Conventional Commits，例如：

- `feat: ...`
- `fix: ...`
- `refactor: ...`
- `docs: ...`

## Git Tag 规则

如果本次变更触发版本升级，且用户明确要求创建版本 tag，则 tag 固定为：

- `vX.Y.Z`

其中 `X.Y.Z` 必须与主版本源一致。

不要创建与主版本源不一致的 tag，也不要混用 `release/X.Y.Z`、`X.Y.Z-release` 等其他格式。

## 同步范围

当主版本源升级后，继续检查以下当前有效入口是否需要同步：

- `stockTrade/backend/app/main.py` 中 FastAPI 的 `version`
- `stockTrade/frontend/package.json` 中的 `version`
- `stockTrade/frontend/src/views/Settings.vue` 中对外展示的版本文本

同步原则：

- 会影响接口文档、运行时元信息、页面展示时，必须同步
- 仅存在于历史文档、旧方案、归档记录中的版本文本，默认不同步
- 如果用户要求创建 release commit 或 git tag，其版本号也必须与主版本源保持一致

## 冲突处理

如果发现多个当前有效入口的版本号不一致，按以下顺序处理：

1. 先确认 `stockTrade/backend/app/__init__.py` 中的 `__version__`
2. 以该值作为当前系统正式版本
3. 将其他仍在使用的有效入口统一到该版本
4. 不为统一版本号而批量回改历史归档文档

## 输出要求

当本次改动涉及版本号处理时，你在结果说明中应明确：

- 是否升级了版本号
- 升级了哪一级别
- 更新了哪些有效入口
- 是否需要 release commit
- 是否需要 git tag
- 哪些历史文档未同步以及为什么未同步

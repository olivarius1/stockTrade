# 版本迭代 AI Rule 设计说明

## 目标

为 `stockTrade` 系统补充一条可执行的 AI rule，用于在日常修改中维护系统版本号，避免版本号遗漏、多个位置不一致，或为了小改动做大范围无意义同步。

## 适用模式

本规则采用轻量模式。

轻量模式的核心原则：

- 维护一个主版本源作为系统正式版本号
- AI 默认只强制维护主版本源
- 其他位置仅在本次改动确实影响对外展示或用户认知时再同步

## 主版本源

当前建议将后端文件 `stockTrade/backend/app/__init__.py` 中的 `__version__` 作为系统主版本源。

原因：

- 它是后端代码内的明确版本入口
- 便于 AI 在代码修改时稳定定位
- 比分散在页面文案或说明文档中的版本文本更适合作为权威版本

## 版本号格式

版本号统一采用语义化版本格式：

- `MAJOR.MINOR.PATCH`
- 示例：`1.3.0`、`1.3.1`

不使用日期版、四段版或带额外前缀的主版本格式。

## 触发规则

以下变更应检查是否需要升级版本号：

- 新增用户可感知功能
- 修改对外接口行为
- 调整部署方式、运行方式或系统配置口径
- 修复真实缺陷且会影响使用结果

以下变更默认不强制升级版本号：

- 纯注释修改
- 纯格式调整
- 无行为变化的重构
- 仅内部草稿文档更新

## 升级策略

- `patch`：Bug 修复、兼容性修正、用户可感知的小修复
- `minor`：新增功能、页面、接口、配置项
- `major`：不兼容变更、重大架构调整、需要迁移成本的变化

如果需求未明确指定版本号，AI 应按上述策略自行判断，并在结果说明中简要交代原因。

## Git 提交与 Tag 规则

当本次变更触发版本升级时，Git 相关约定如下：

- release commit message 使用：
  - `chore(release): bump version to X.Y.Z`
- git tag 使用：
  - `vX.Y.Z`

普通开发提交不使用 release commit message，继续沿用 Conventional Commits，例如：

- `feat: ...`
- `fix: ...`
- `refactor: ...`
- `docs: ...`

只有在用户明确要求提交 commit 或创建 tag 时，AI 才执行对应 Git 操作；但规则本身需要约定好命名格式。

## 同步策略

当主版本源升级后，AI 需要进一步检查以下位置是否需要同步：

- FastAPI 应用版本声明
- 前端 `package.json` 版本号
- 系统设置页或页面中的对外展示版本文本

同步原则：

- 如果这些位置直接对用户展示，或会影响运行时接口描述，应同步
- 如果只是历史设计文档、旧计划文档中的示例文本，默认不做全量回写

当版本升级且用户要求执行发布动作时，AI 还应保证：

- release commit message 中的版本号与主版本源一致
- git tag 中的版本号与主版本源一致

## 冲突处理

如果 AI 在修改过程中发现多个版本号不一致，应：

1. 先确认主版本源
2. 以主版本源为准统一当前有效版本
3. 仅同步当前仍在使用的代码和展示入口
4. 不强制回改历史归档文档

## 预期产出

最终写入 `.trae/rules/版本迭代.md` 的规则应具备以下特征：

- 简洁
- 可执行
- 明确主版本源
- 明确何时升级、何时不升级
- 明确同步边界

## 一、执行摘要

当前项目已经达到“可运行的演示版 / POC”状态，但尚未达到需求文档所描述的“完整闭环估值系统”状态。

已确认落地的内容：

- 前后端、Redis、Celery、Docker Compose 已可本地运行
- 登录、自选股管理、模型列表、定时配置页、基础估值计算链路可用
- 后端单元测试通过，当前容器运行稳定

尚未 100% 落地的关键内容：

- 数据服务闭环未完成：股票搜索为空实现，K 线历史未落库，财报抓取未实现
- 定时任务仅完成“配置存库 + 手动触发”，未完成“配置驱动调度执行”
- 需求与实现不一致：认证方式、接口设计、模型/因子管理能力、数据来源策略均存在偏差
- 前端多个页面为展示型实现，缺少对真实业务状态、错误边界和配置持久化能力的完整支撑

结论：

- 项目状态等级：`黄灯`
- 交付成熟度：`可演示，不可视为生产可交付`
- 建议先关闭 P0 缺口，再进入新增需求或体验打磨阶段

## 四、原始需求文档审计结论

总体判断：

- 需求方向是合理的，目标用户、核心能力和系统边界大体清晰
- 但文档混合了“POC 范围”“中长期规划”“预留能力”，缺少明确的当前交付边界
- 部分需求描述存在逻辑冲突、边界模糊或可执行性不足，已经开始反向影响实现一致性

### 4.1 需求问题清单

| 编号 | 问题类型 | 需求问题 | 影响 | 建议 |
| --- | --- | --- | --- | --- |
| R1 | 逻辑矛盾 | 文档写“简单密码保护、单用户模式、检查 Cookie”，但安全章节又写“JWT 或 Session Cookie”，当前实现为 JWT + 用户表 | 认证方案摇摆，导致前后端、接口、数据模型不统一 | 明确 V1 只做“单管理员 + JWT”或“单密码 + Cookie”二选一 |
| R2 | 逻辑矛盾 | 模型管理需求写“查看/配置估值模型和因子”，但 API 与页面实际更像“只读展示” | 用户预期会认为可以编辑模型，实际无法操作 | 将 V1 改为“模型查看”，把“模型配置”下放到 V1.1 |
| R3 | 逻辑矛盾 | 设计文档中的模型因子接口、API 路由与当前实现不完全一致，如因子接口写 `/api/factors`，实现为 `/api/models/factors` | 需求、前端、后端的契约不稳定 | 生成唯一接口清单，并作为验收基线 |
| R4 | 边界模糊 | “AI 深度分析因子”只给了权重建议，没有定义模型来源、Prompt、超时、失败回退、费用控制 | AI 因子无法稳定落地，容易成为伪功能 | 为 AI 分析补全输入、输出、缓存、超时与熔断策略 |
| R5 | 边界模糊 | “财报数据优先 MCP，备用公开 API”未指定具体提供方、字段映射、权限与刷新策略 | 数据服务无法工程化，落地成本被低估 | 明确主数据源、备用源、字段字典和刷新 SLA |
| R6 | 边界模糊 | “历史日 K 线持久化、当天盘中实时请求”未明确回补窗口、异常容错、停牌处理、复权口径 | K 线存储与估值结果口径难以统一 | 补充日线回补范围、复权口径、停牌处理规则 |
| R7 | 可执行性不足 | 定时任务仅描述“用户可配置”，但没有说明 Cron 校验、时区、热更新、失败告警、幂等策略 | 当前只能做“配置展示”，难做可靠自动化 | 为调度能力增加配置生效机制和任务状态定义 |
| R8 | 可执行性不足 | 估值分与百分位有公式，但没有说明行业基准、样本窗口、空数据时如何剔除与重归一化 | 算法实现容易简化，结果解释性不足 | 给出评分样例、窗口定义和缺失数据规则 |
| R9 | 范围混杂 | “数据导出、多用户、移动端、量化策略集成”等预留需求直接出现在主文档 | 让当前版本边界模糊，压缩核心功能资源 | 将预留项迁移到 roadmap，不纳入 V1 完成定义 |
| R10 | 验收缺失 | 文档没有为“完成”提供可验证标准，如接口返回、页面可用性、数据时效、任务成功率 | 任务很难被审计为 100% 落地 | 为每个模块补充验收标准与测试样例 |

### 4.2 对原始需求的总体优化建议

建议将需求重新切分为三层：

1. `V1 必做`：登录、自选股、单股估值、历史曲线、手动批量计算、基础定时任务、Docker 本地部署
2. `V1.1 增强`：模型配置编辑、AI 因子、财报自动更新、搜索联想、结果导出
3. `Roadmap`：多用户、移动端、邮件通知、量化策略集成

## 六、项目优化实施方案

### 6.1 第一阶段：关闭 P0 缺口（建议 3-5 天）

目标：把项目从“演示可跑”提升到“核心闭环可验收”

建议动作：

1. 实现 `search_stock()`，接入稳定的股票搜索源
2. 完成财报抓取与字段映射，至少支撑 `roe`、`dividend`、`eps`
3. 实现 K 线历史回补与落库，建立数据刷新策略
4. 补齐缺失接口：`logout`、手动计算接口、统一因子接口路由
5. 将调度配置正式绑定 Celery Beat 生效逻辑
6. 去除缓存中的 `eval()`，改为 JSON 序列化

阶段交付物：

- 数据层闭环可运行
- 搜索、自选、估值、批量计算、定时执行全链路联调通过
- 接口文档与实现一致

### 6.2 第二阶段：提升可维护性与可解释性（建议 3 天）

目标：把当前实现从“堆功能”调整为“可维护系统”

建议动作：

1. 引入数据库迁移工具
2. 拆分 API 层重复逻辑，收敛到 service 层
3. 为估值结果补充数据来源、时间戳、失败回退说明
4. 明确模型/因子是“只读”还是“可编辑”，并同步前后端
5. 完善错误码与前端提示语

阶段交付物：

- 统一服务边界
- 统一接口契约
- 页面错误与状态表达更准确

### 6.3 第三阶段：补齐验收与发布能力（建议 2-3 天）

目标：让项目具备稳定交付条件

建议动作：

1. 补 API 集成测试：登录、搜索、自选股、估值、调度
2. 补前端 E2E：登录、首页、自选股、报告页
3. 为 Docker Compose 增加健康检查与运行说明
4. 建立版本化验收清单和回归基线

阶段交付物：

- 冒烟测试清单
- 回归测试基线
- 可复用的本地 / 预发布发布说明

## 七、最终判断

### 7.1 当前项目状态

- 工程状态：`可运行`
- 需求状态：`部分落地`
- 交付状态：`可演示，不可宣称全部完成`

### 7.2 管理层视角建议

当前不建议将项目定义为“已完成上线准备”，更适合定义为：

`核心骨架完成，正在补齐数据闭环与调度闭环的开发中版本`

### 7.3 审计建议结论

优先处理顺序应为：

1. 数据服务闭环
2. 定时任务真调度
3. 接口与需求对齐
4. 测试与验收补齐
5. 用户体验优化

在上述 P0 项全部关闭之前，不建议继续扩展 AI 因子、多用户、移动端等新增范围。
